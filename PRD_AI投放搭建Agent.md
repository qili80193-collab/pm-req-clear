## 1. 背景与目标

### WHY

公司内部产研运团队（主要是运营）用创量平台手动搭建投放计划，操作繁琐且耗时长。痛点是页面手动创建，效率低、重复劳动多。做完后，运营只需用自然语言下指令（素材条件、数量、预算、出价等），AI Agent 自动完成计划的创建和组合，人工确认后直接推送媒体。

### 范围

| 分期 | 内容 |
|------|------|
| **v1** | 自然语言选素材 → 下指令 → Agent 生成逐条方案 → 人工确认 → 推送头条+快手 |
| **v2** | 引用历史计划修改、定时/周期投放、更多平台 |
| v2 启动条件 | v1 核心闭环跑通，采纳率 ≥95% |

## 2. 前端展示

### 2.1 交互位置

Agent 对话界面嵌入内部投放系统，作为"智能搭建"入口，替代现有的手动创建页面流程。

### 2.2 交互流程

**单意图**（如纯搭建）：

```
① 用户输入 → ② 路由判断意图 → ③ 对应 Agent 处理 → ④ 返回结果
```

**多步骤工作流**（如分析→优化→执行）：

```
① 用户输入
    ↓
② 路由识别为多步骤工作流 → 拆解为步骤序列
    ↓
③ 按序调度：
   Step 1: 查询 Agent → 返回数据
   Step 2: 分析 Agent → 输出建议（用户可追问细化）
   Step 3: 搭建 Agent → 执行调整 → 展示结果
```

### 2.3 快捷操作按钮

对话输入框上方固定展示 3 个快捷按钮，点击后自动填入触发指令：

| 按钮 | 点击后 Agent 响应 |
|------|-----------------|
| 📊 查消耗 | "你想查什么数据？请告诉我时间范围和具体指标。例如：查本周的消耗和展示数据 / 查上周的转化率和 CPA" |
| 🔍 选素材 | "你想用什么条件筛选素材？时间范围、标签、还是按表现数据排序？" |
| 🏗️ 快速搭建 | "好的，我来帮你搭建计划。投哪个平台？建多少条？预算和出价方式？" |

操作路径：

**📊 查消耗：** 点击 → Agent 询问查询范围 → 用户描述 → 查询 Agent 拉数据 → 展示结果
**🔍 选素材：** 点击 → Agent 询问筛选条件 → 用户描述 → 查询 Agent 匹配素材 → 展示列表 → 用户确认
**🏗️ 快速搭建：** 点击 → Agent 逐个追问参数 → 用户回答 → 搭建 Agent 生成方案 → 确认 → 推送

### 2.4 历史会话

左侧会话列表展示历史对话记录，按时间倒序排列。点击可查看完整的对话内容，不可继续编辑。

## 3. 核心规则与逻辑

### 3.1 系统架构总览

```
用户输入
    ↓
┌─ 路由 Agent（小 LLM）──────────────────────────┐
│  1. 意图识别（单意图 vs 多步骤工作流）            │
│  2. 工作流拆解（多步骤时拆分步骤序列）            │
│  3. 会话阶段管理（维护执行状态）                  │
│  4. 步骤调度与数据传递                          │
│  5. 中断处理（追问/回退/修改）                   │
└────────────────────────────────────────────────┘
    ↓           ↓             ↓
查询 Agent   分析 Agent    搭建 Agent
```

### 3.2 路由 Agent 设计

路由 Agent 是一个独立的小 LLM 调用，负责识别用户意图并协调执行流程。

#### 3.2.1 路由输入

```
用户自然语言 + 当前会话上下文
```

#### 3.2.2 路由输出格式

```json
{
  "type": "single" | "workflow",
  "intent": "analyze_and_optimize",

  // 单意图：直接分发
  "target_agent": "query",

  // 多步骤工作流：定义执行序列
  "workflow": [
    {
      "step_id": 1,
      "agent": "query",
      "task": "查询本周投放数据",
      "params": {"time_range": "本周", "metrics": ["消耗","展示","点击","转化"]},
      "depends_on": null
    },
    {
      "step_id": 2,
      "agent": "analyze",
      "task": "分析数据异常并给出优化建议",
      "params": {"focus": "异常检测"},
      "depends_on": [1]
    },
    {
      "step_id": 3,
      "agent": "builder",
      "task": "按建议执行调整",
      "params": {"mode": "apply_suggestions"},
      "depends_on": [2]
    }
  ]
}
```

#### 3.2.3 意图识别规则

| 用户输入 | 类型 | 路由结果 |
|---------|------|---------|
| "建 100 条计划预算 5 万" | single | target_agent: builder |
| "查上周消耗数据" | single | target_agent: query |
| "分析下为什么这条计划跑量差" | single | target_agent: analyze |
| "预算改 8 万" | single | target_agent: builder (modify) |
| "分析本周数据，给建议，按建议调出价" | workflow | 三步工作流 |
| "查上周数据，效果差的调低出价" | workflow | 两步工作流 |
| "看一下投放情况"（歧义） | single | target_agent: query（默认） |

#### 3.2.4 会话阶段管理

路由 Agent 维护会话的阶段状态，支持多轮跨 Agent 的连续对话：

**阶段定义：**

| 阶段 | 含义 | 下一个可能的阶段 |
|------|------|----------------|
| idle | 空闲，等待新需求 | querying / analyzing / building |
| querying | 查询 Agent 执行中 | idle / analyzing / building |
| analyzing | 分析 Agent 执行中 | idle / building |
| building | 搭建 Agent 执行中 | idle / modifying |
| modifying | 修改参数执行中 | idle / building |
| completed | 当前任务完成 | idle（回到待命）|

**阶段流转示例：**

```
idle → querying → analyzing → building → modifying → completing → idle
  (查数据)  (分析给建议) (执行调整)  (再修改)     (完成)
```

**各阶段行为：**

| 当前阶段 | 用户输入特征 | 路由行为 |
|---------|------------|---------|
| querying | "展开看看"、"哪些指标异常" | 留在当前 Agent 细化 |
| querying | "分析一下为什么" | 推进到 analyzing 阶段 |
| analyzing | "按建议执行" | 推进到 building 阶段 |
| analyzing | "这个建议不行，换个思路" | 回到当前 Agent 重新分析 |
| building | "预算改 8 万" | 推进到 modifying |
| building | "确认创建" | 推进到 completed |
| any | "回到上一步" | 回退到上一阶段 |
| any | "不做了" | 回到 idle |
| completed | "再建一批" | 回到 idle 重新开始 |

### 3.3 工作流识别与编排

#### 3.3.1 识别条件

路由 Agent 判断"一句话是否包含多步骤工作流"的依据：

```
用户输入中有多个操作动词，且后一步依赖前一步的输出。

特征词："然后"、"再"、"并"、"按建议"、"根据结果"
依赖链：查→分析→调整、查→改、分析→执行
```

#### 3.3.2 工作流执行流程

```
路由识别为工作流 → 拆解 steps
    ↓
Step 1 出队 → 调度对应 Agent → 执行 → 返回结果
    ↓
中间结果写入 session.context
    ↓
展示给用户（支持追问细化）
    ↓
用户确认/进入下一步
    ↓
Step 2 出队 → 调度对应 Agent → 读取上一步结果 → 执行 → 返回结果
    ↓
...
    ↓
全部完成 → 汇总 → 回到 idle
```

#### 3.3.3 用户交互点

工作流不是黑盒。每一步完成后展示给用户，用户可：

| 用户行为 | 路由响应 |
|---------|---------|
| 直接说"继续" | 推进到下一步 |
| 追问中间结果 | 留在当前 Agent 细化回答 |
| "不对，换个方向分析" | 重新调度当前 Agent |
| "跳过分析直接执行" | 跳过指定步骤 |
| "回到上一步" | 回退，重新调度前面的 Agent |

### 3.4 搭建 Agent 内部流程

```
路由分发"搭建"任务 → 搭建 Agent 开始
    ↓
解析自然语言 → 查询素材库 → 展示匹配结果 → 用户确认
    ↓
生成搭建方案 → 展示逐条预览 → 用户确认
    ↓
推送媒体（头条+快手并行） → 返回结果
```

### 3.5 自然语言解析规则（搭建 Agent）

**输入：** 用户自然语言
**输出：** 结构化参数

| 字段 | 解析方式 | 示例 |
|------|---------|------|
| 时间范围 | 相对时间转具体日期 | "最近两周" → 近 14 天 |
| 排序条件 | 字段 + 排序方向 + 截断 | "点击率前5" → sort_by:ctr, limit:5 |
| 数值提取 | 数字直接提取 + 单位换算 | "5万" → 50000 |
| 平台 | 提取平台名 | "头条""快手""都投" |

**缺参处理（必须追问，不默认填值）：**

| 必需参数 | 追问示例 |
|---------|---------|
| 平台 | "投哪个平台？头条还是快手？" |
| 数量 | "建多少条计划？" |
| 总预算 | "总预算是多少？" |
| 出价方式 | "出价方式是什么？oCPM 还是 CPA？" |

### 3.6 素材筛选规则

| 筛选维度 | 匹配方式 | 示例 |
|---------|---------|------|
| 上传时间 | 解析为具体时间范围 | "最近两周" → 近14天 |
| 标签 | 精确匹配素材标签 | "标签为 agent" |
| 点击率 | 排序 + 截断 | CTR desc limit 5 |
| 素材类型 | 按类型过滤 | media_type=video |
| 组合筛选 | 多条件 AND | "最近两周"+"标签 agent"+"点击率前5" |

**边界规则：**
- 匹配结果为空 → 告知用户无结果，建议放宽条件
- 匹配结果 < 计划组数 → 按实际素材组合，告知受限
- 文案库不足 100 条 → 按实际文案数组合，重复使用

### 3.7 计划生成规则

**素材组合方式：**
- 视频 × 文案交叉组合
- 同组计划参数相同，仅名称编号递增
- 各组计划数按组合数均分（有余数时分配给首组）

**各平台差异化参数：**

| 参数 | 头条 | 快手 |
|------|------|------|
| 计划名称前缀 | `tt_` | `ks_` |
| 定向年龄 | 18-40 | 18-45 |
| 素材规格 | 视频 16:9 | 视频 9:16 或 1:1 |
| 监测链接参数 | utm_source=toutiao | utm_source=kuaishou |

### 3.8 推送规则

- 头条和快手独立推送，不互相阻塞
- 推送超时阈值：30s
- 失败重试：最多 3 次，间隔 5s/10s/20s
- 部分成功时：成功计划已创建，失败计划列原因
- 用户可对失败计划批量重试

### 3.9 边界处理

| 场景 | 处理方式 |
|------|---------|
| 素材匹配结果为空 | 告知用户无符合条件的素材，建议放宽条件 |
| 预算不够分 | 自动提示，建议增加预算或减少计划数 |
| 推送部分失败 | 已成功的保留，失败列表列出原因，支持重试 |
| 媒体 API 限频/超时 | 自动重试 3 次，间隔递增，仍失败则通知用户 |
| 用户对话中断 | 保存当前进度，下次可继续 |
| 文案库不足 | 按实际文案数组合，告知用户 |
| 工作流中间步骤用户追问 | 留在当前 Agent 细化，不推进下一步 |
| 用户要求跳过某步骤 | 从工作流中移除该步骤，推进到下一步 |
| 用户要求回退 | 回退到上一阶段，重新调度之前的 Agent |

## 4. 数据层

### 4.1 投放报表数据源

查询 Agent 可查询的字段：

| 字段 | 说明 | 示例 |
|------|------|------|
| date | 日期 | 2026-06-10 |
| account_id | 账户 ID | acc_001 |
| campaign_id | 计划 ID | cmp_1001 |
| campaign_name | 计划名称 | 新品推广_001 |
| platform | 投放平台 | 头条 / 快手 |
| cost | 花费（元） | 12500.00 |
| impression | 展示次数 | 2847239 |

**查询维度：** 按时间范围（本周/上周/近7天/自定义）、按平台、按账户/计划

### 4.2 素材库

素材基本字段：

| 字段 | 说明 | 示例 |
|------|------|------|
| material_id | 素材 ID | m_001 |

| type | 素材类型 | video / image |
| tags | 标签列表 | ["agent", "横屏16:9"] |
| create_time | 上传时间 | 2026-06-10 |
| duration | 视频时长（秒） | 15 |
| status | 状态 | active / paused / rejected |

素材表现数据：

| 字段 | 说明 | 示例 |
|------|------|------|
| cost | 该素材累计花费 | 82300.00 |
| impression | 该素材累计展示 | 15300000 |

**筛选维度：** 上传时间范围 / 最近使用时间 / 标签（AND） / 按 CTR/CPA/次留排序取前 N / 素材类型 / 状态

### 4.3 媒体平台接口

## 5. 埋点需求

### 事件清单

| # | 事件名 | 触发时机 | 触发方式 |
|---|--------|---------|---------|
| 1 | agent_build_started | 用户输入搭建指令 | Agent 捕捉 |
| 2 | agent_material_confirmed | 用户确认素材 | 按钮点击 |
| 3 | agent_material_modified | 用户修改筛选条件 | 按钮点击 |
| 4 | agent_plan_confirmed | 用户确认搭建方案 | 按钮点击 |
| 5 | agent_plan_modified | 用户修改方案参数 | 按钮点击 |

### 埋点用途

## 6. 验收标准

| 场景 | 用户操作 | 预期结果 |
|------|---------|---------|
| 从零搭建 | 输入完整指令（素材条件 + 参数） | Agent 展示素材 → 用户确认 → 生成预览 → 推送成功 |
| 参数调整 | 看完预览说"改预算为 X" | 预览更新，其余不变 |
| 素材不足 | 条件过窄无匹配 | Agent 提示无结果并建议放宽条件 |
| 推送部分失败 | 确认创建后部分失败 | 显示成功数量 + 失败列表，支持重试 |
| 缺参追问 | 只说"建一些计划" | Agent 追问具体参数，不默认填值 |
| 文案库不足 | 计划数 > 文案库规模 | Agent 提示并按实际文案数组合 |

## 7. 衡量计划

| 验证问题 | 获取方式 | 节奏 |
|---------|---------|------|
| 创建耗时是否降到 3-5 分钟？ | 埋点：时间差 | 上线 1 周后评估 |
| Agent 采纳率是否 ≥95%？ | 埋点：直接确认 vs 修改后确认 | 上线 1 周后评估 |
| 推送成功率多高？ | 埋点：platform success_count | 持续监控 |
| 跑量效果是否下降？ | 对比 Agent 创建 vs 手动创建的 7 日跑量 | 上线 1 周后评估 |
